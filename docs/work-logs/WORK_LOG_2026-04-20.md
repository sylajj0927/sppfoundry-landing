BOOM should currently remain as a mature tool on its own subdomain:

\- \`https://boom.sppfoundry.co/\`

Reason:  
\- current agreed SPP Foundry structure treats mature tools like BOOM and BiteMap as subdomain products  
\- \`/labs/\` is primarily for live MVPs / smaller experimental tools  
\- moving BOOM again right now would create unnecessary migration risk and duplicate work

\#\# 6\. Relationship to old Netlify setup  
The old Netlify BOOM project was still connected to GitHub continuous deployment.

This created two important observations:  
1\. pushing changes to GitHub could still trigger Netlify deploy activity  
2\. this is no longer the desired long-term production path for BOOM

Because of that, Netlify is now considered legacy infrastructure for BOOM, not the preferred active deployment path.

\#\# 7\. Cleanup action completed  
The GitHub repository connection inside the old Netlify BOOM project was unlinked.

Status after change:  
\- Netlify BOOM project no longer auto-deploys from GitHub  
\- future BOOM Git pushes should no longer keep feeding an unnecessary parallel Netlify pipeline

A follow-up question was reviewed: whether GitHub-side Netlify permissions must also be removed.

Decision:  
\- not required tonight for site stability  
\- can be treated as an optional cleanup step later

Why:  
\- unlinking the repository inside the Netlify project already stops the direct continuous deployment connection for this project  
\- GitHub app permission cleanup is secondary and can be done later when doing broader Netlify reduction

\#\# 9\. Main site relationship  
The main SPP Foundry site should later include BOOM as a discoverable entry / work link, but BOOM itself should continue living on its own production subdomain.

\#\# 11\. What was intentionally not done  
To reduce risk, today did \*\*not\*\*:  
\- move BOOM into \`/labs/\`  
\- delete old Netlify BOOM project immediately  
\- delete old Netlify \`sppfoundry.co\` project immediately  
\- remove GitHub-wide Netlify app permissions  
\- make BOOM feature/content changes

\#\# 12\. Lessons learned  
A domain can already be publicly working on Cloudflare while old Netlify project links still exist in the background.  
So migration cleanup should be done in layers:

1\. confirm new production domain works  
2\. stop old auto-deploy linkage  
3\. keep legacy project temporarily until stability is confirmed  
4\. remove old platform access/permissions only after structure is fully clear

This is safer than deleting old deployment infrastructure too early.
